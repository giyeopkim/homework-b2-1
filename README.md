### 간단하게 명세된 기능
#### add (거래 추가)
- 입력/요청: 대화형 입력(날짜, 타입, 카테고리, 금액, 메모/태그)
- 출력/화면: 저장 성공 메시지 + 생성된 거래 id
- `add`실행 후 대화형으로 필드를 입력 받아 거래를 저장
- category는 등록된 목록에 존재해야함. (없으면 안내 후 재입력/등록 유도)
- 저장 완료 시 생성된 `id`를 사용자에게 출력해줌.
#### list
- 입력/요청: -limit 등 옵션
- 출력/화면: 최신순 거래 리스트(스트리밍 처리)
- `list`는 최신 순으로 거래를 출력
- `--limit N` 옵션을 지원해야함(기본값 제공)
- 파일 전체를 한 번에 로드하지 않고, 제너레이터 기반 스트리밍 처리로 구현
#### search
- 입력/요청: -from/--to, -category, -type, -q, -tag
- 출력/화면: 조건에 맞는 거래 리스트(최신순)
- 조건 기반 검색을 지원
    -기간: `--from`, `--to`
    -카테고리: `--category`
    -타입: `--type`
    -메모 키워드: `--q`
    -태그: `--tag`
- 검색 결과는 최신순으로 출력
- 제너레이터 기반 스트리밍 처리
#### summary
- 입력/요청: -month YYYY-MM, -top N
- 출력/화면: 총수입/총지출/잔액 + 카테고리별 지출 TOP N
- `summary --month YYYY-MM` 입력을 받아 해당 월의 요약을 출력
- 출력 항목
    - 총 수입, 총 지출, 잔액(총수입-총지출)
    - 카테고리별 지출 합계 TOP N(-top 옵션 지원)
- 데이터가 없는 달은 "데이터 없음"을 명확히 출력
#### budget
- 입력/요청: budget set --month YYYY-MM --amount 금액
- 출력/화면: 저장 성공 메시지 + summary에서 예산 사용률/초과 경고
- 입력/요청으로 월 예산을 작성
- `summary` 실행 시 예산이 설정되어 있으면
    - 예산 대비 사용룰(%)
    - 초과 여부와 경고 문구 출력
- 예산 데이터도 영구 저장
#### category
- 입력/요청: category add/list/remove
- 출력/화면: 카테고리 목록/추가/삭제 결과(사용 중 카테고리 처리 포함)
- 카테고리 삭제시, 해당 카테고리를 사용하는 내역이 존재하면 삭제를 막거나 대체 카테고리를 요구해야 함.
#### update
- 입력/요청: -id 기반(옵션 방식 또는 대화형 중 1개로 고정)
- 출력/화면: 수정 성공/실패 메시지(없는 id 처리 포함)
- `update --id <id> [--data ...] [--type ...] [-category ...] [--amount ...] [--memo ...] [--tags ...]`
- 존재하지 않는 id는 "없는 데이터"로 처리하고 사용자 메시지 출력
- 파일 기반 저장에서 전체 재작성/임시 파일/원자적 교체 등 안정성을 고려.
#### delete
- 입력/요청: delete --id <id>
- 출력/화면: 삭제 성공/실패 메시지(없는 id 처리 포함)
- `delete --id <id>`로 특정 거래 삭제 가능
- 존재하지 않는 id는 "없는 데이터"로 처리하고 사용자 메시지 출력
- 파일 기반 저장에서 전체 재작성/임시 파일/원자적 교체 등 안정성을 고려.
#### expoert / import
- 입력/요청: import --from <csv>, export --out <csv> --month ... 또는 -from/--to
- 출력/화면: 처리 건수 출력 + CSV 파일 생성/반영 확인
- `import --from`으로 거래를 일괄 등록
- `export --out`으로 조건에 맞는 거래를 csv로 저장
- `export--month YYYY-MM`
- `export--from YYYY-MM-DD --to YYYY-MM-DD`
- `import/export`는 아래 CSV 최소 스키마를 고정한다.

    | column | required | 설명 |
    | --- | --- | --- |
    | date | Y | YYYY-MM-DD |
    | type | Y | income / expense |
    | category | Y | 등록된 카테고리 |
    | amount | Y | 양수 정수 |
    | memo | N | 문자열 |
    | tags | N | 쉼표(,) 구분 문자열 |
    | 공통: UTF-8, 헤더 포함 |   |   |


#### save/load
- 3개 이상의 파일로 데이터 영구 저장 (transaction / categories / budgets)
- file type : JSONL


### 실행 및 입력 방식
#### 대화형 구조를 가질 것.
- `add` 실행 시 날짜/타입/카테고리/금액 등을 `input()`으로 순차 입력
- 단, `search`, `list`, `summary`, `export`, `import`, `delete`는 옵션 인자 방식도 허용.
- `update`는 "옵션 방식 또는 대화형" 중 하나를 선택해도 됨. but, 문서에 명확히 고정.
### 옵션 표기는 --로 통일
- ex) `--help`, `--limit`, `--from`, `--to`, `--month`

### 모델 구조
#### 트랜잭션은 최소 아래 필드를 포함.
- `id`(유일), `type`(income/expense), `date`(YYYY-MM-DD), `amount`(양수), `category`, `memo`(선택), `tags`(선택)
#### `dataclass` 또는 그에 준하는 구조
#### `Transaction`, `TransactionRepository`, `BudgetStore`, `CategoryStore`, `BudgetService` 등의 클래스를 사용
#### Input Validation
- 날씨 형식 오류, 음수/0 금액, 허용되지 않은 type, 존재하지 ㅇ낳는 category 등에 대해서 검증 처리가 필요
- 재입력 요구 또는 오류 메시지 출력

### 저장 정책
#### export/import 저장 형식
- CVS 타입을 사용
#### 영구저장 목적의 파일 입출력 파일 형식
- JSONL 타입을 사용
#### 3개 이상의 영구 저장 파일
- `./data/transcations.jsonl`, `./data/categories.jsonl`, `./data/budgets.jsonl`
#### 저장 디렉토리를 바꾸는 옵션이 존재해야함.
#### 초기 실행
- 파일이 없으면 자동 생성 또는 초기화 안내 메시지 출력
- 카체고리 파일이 비여 있으면 `category add`를 먼저 하도록 안내하고 그 후 `add`를 막음
